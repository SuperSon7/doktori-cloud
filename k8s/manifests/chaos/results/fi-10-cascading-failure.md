## FI-10: 연쇄 장애 (Cascading Failure)

- 실행 시각: 2026-03-25 16:47:41 ~ 16:50:44
- 부하 상태: load 실행 중 (API CPU 105%, 2 replicas)
- 결과: ⚠️ 연쇄 장애 대신 **의도치 않은 스케일다운 발생**

### 관측값

| 지표 | 기대 | 실측 |
|------|------|------|
| DB 500ms 지연 | CPU 폭증 → 커넥션 풀 고갈 | ❌ **CPU 24%로 급감** |
| HPA 반응 | 스케일아웃 | ❌ **4→2 스케일다운** (CPU 60% 미만) |
| 연쇄 장애 | DB→커넥션→5xx | 관찰 안 됨 — CPU 기반 HPA가 "안정"으로 판단 |

### 타임라인

| 시간 | CPU | Replicas | 상태 |
|------|-----|----------|------|
| BEFORE | 105% | 2 | 부하 중 |
| +30s | 98% | 4 | 일시적 스케일아웃 |
| +60s | 87% | 2 | 스케일다운 시작 |
| +90s | **24%** | 2 | **DB 지연으로 CPU 급감** |
| +120~180s | 24~57% | 2 | 낮은 CPU 유지 |

### 핵심 발견 (FI-3과 동일)

**DB 지연 시 CPU가 오히려 떨어지는 패턴 재확인:**
- DB 응답 대기 중 스레드가 idle → CPU utilization 감소
- HPA는 CPU만 보고 "부하가 줄었다"고 판단 → 스케일다운
- **실제로는 응답 latency 500ms+ 증가 → 사용자 경험 악화**
- 하지만 CPU HPA는 이를 감지 못 함

### 시사점

1. **CPU 기반 HPA는 DB 지연 장애를 감지 못 하는 맹점** — FI-3에서 발견, FI-10에서 재확인
2. 연쇄 장애(커넥션 풀 고갈 → 5xx)까지는 안 갔지만, **HPA가 역방향 반응**(스케일다운)하는 더 나쁜 상황
3. latency 기반 스케일링 또는 HikariCP pending 기반 alert가 필요

### 후속 조치
- API HPA에 latency 기반 메트릭 추가 검토 (Prometheus Adapter 필요)
- 또는 HikariCP pending > 0일 때 alert으로 수동 대응
- Circuit Breaker 패턴 도입 (DB 응답 1초 초과 시 fast-fail)

## 분석

### WebSocket이 API 성능을 크게 악화시킨 이유

1. **Chat Pod + API Pod가 같은 워커 노드에서 리소스 경합**
   - Chat Pod 2 replica + API Pod 2 replica = 4 Pod가 워커 노드 4대에 분산
   - WebSocket은 장시간 연결을 유지하므로 메모리 + 스레드를 지속 점유
   - Chat Pod가 리소스를 먹으면서 같은 노드의 API Pod 성능 저하

2. **WebSocket 100 VU × 34초 평균 세션 = 상시 ~50 동시 연결**
   - 각 연결이 STOMP 메시지를 주기적으로 전송
   - RabbitMQ 메시지 브로커 부하 추가

3. **총 VU가 줄었는데도 (300→300) 성능이 훨씬 나빠짐**
   - WebSocket 연결은 HTTP 요청보다 리소스 소모가 큼
   - 연결 유지 + 메시지 브로드캐스트 비용

### 개선 방향

| 우선순위 | 개선 | 기대 효과 |
|---------|------|----------|
| P0 | Chat Pod에 리소스 request/limit 조정 | API Pod와의 경합 완화 |
| P0 | Pod anti-affinity 강화 — Chat/API를 다른 노드에 배치 | 리소스 격리 |
| P1 | HPA maxReplicas 증가 | 부하 분산 |
| P1 | WebSocket 연결 수 제한 (Chat 서비스 레벨) | 과도한 연결 방지 |
| P2 | 워커 노드 스펙 업그레이드 (t4g.large → t4g.xlarge) | 전체 리소스 증가 |
